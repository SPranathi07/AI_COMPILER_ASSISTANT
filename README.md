## Project Overview

The AI Compiler Assistant is a mini C compiler extended with an AI layer that provides plain English error explanations, syntax auto-completion, optimization hints, and security analysis. Traditional compilers give cryptic error messages that beginners cannot understand. This project solves that problem by integrating AI-powered diagnostics directly into the compiler front end.

---

## Problem Statement

Traditional compilers translate source code into executable programs while reporting errors during compilation. However, diagnostic messages generated by conventional compilers are often cryptic, lack contextual explanations, and do not provide proactive guidance to developers. This makes it difficult, especially for beginners, to understand syntax errors, optimize code, and avoid insecure coding practices.

The proposed system enhances the compiler front end by providing:
- Syntax auto-completion
- Human-readable error explanations
- Optimization hints
- Security analysis

## Weekly Implementation Summary

### Week 1 — Problem Understanding
Studied compiler architecture including lexical analysis, parsing, AST, and error handling. Identified that traditional compiler diagnostics are cryptic and unhelpful for beginners. Documented the problem statement, motivation, and compiler workflow overview.

### Week 2 — Literature Survey
Reviewed existing tools including GCC, Clang, IntelliSense, GitHub Copilot, and research systems like DeepFix. Identified the gap: no existing tool combines compiler integration, plain English explanation, offline operation, and beginner focus.

| Tool | Technique | Limitation |
|---|---|---|
| GCC | Rule-based | Cryptic messages, no explanation |
| Clang | Static analysis | No intelligent guidance |
| IntelliSense | Heuristics + ML | Limited error explanation |
| Copilot | Transformer | Not compiler integrated |

### Week 3 — Requirements Analysis
Defined functional requirements (tokenization, parsing, error detection, AI explanation, autocomplete) and non-functional requirements (70% accuracy, under 1 second response, cross-platform). Prepared use-case scenarios and SRS document.

### Week 4 — System Architecture and Design
Designed the layered architecture: C compiler front end communicating with Python AI layer via JSON through stdout. Selected C for the compiler (performance, standard), Python for AI (scikit-learn, numpy), and JSON for data exchange.

### Week 5 — Grammar Design
Defined Mini-C grammar covering:
- Data types: `int` `float` `char` `void`
- Keywords: `if` `else` `while` `for` `return`
- Operators: arithmetic, comparison, logical, assignment
- Constructs: functions, arrays, if/else, loops

### Week 6 — Lexer Implementation (`Lexer.c`)
Implemented the lexical analyser that reads source code character by character and produces a stream of typed tokens with line and column positions.

**Key functions:**
- `advance()` — reads one character, tracks line/column
- `peek()` — looks ahead without consuming
- `checkKeyword()` — distinguishes keywords from identifiers
- `nextToken()` — main function returning one token per call
- `tokenTypeName()` — converts token type to string for JSON output

**Token types supported:** 36 types including keywords, identifiers, literals, operators, delimiters, and EOF/UNKNOWN.

### Week 7 — Parser and AST (`Parser.c`)
Implemented a recursive descent parser that checks grammar rules, builds an Abstract Syntax Tree, and reports errors with panic mode recovery. Modified to output JSON errors for Python integration.

**Key features:**
- 18 AST node types
- Operator precedence chain (9 levels)
- PARSE_BINOP macro for 6 operator levels
- Panic mode error recovery
- JSON error output for Python integration

**Syntax errors detected:** missing semicolons, unmatched braces, unmatched parentheses, invalid expressions, invalid assignment targets.

### Week 8 — Dataset Creation (`dataset_builder.py`)
Built an annotated dataset of 50 C code samples for AI model training.

**Dataset statistics:**
- Total samples: 50
- Correct samples: 25 (50%)
- Incorrect samples: 25 (50%)
- Error types: 5 (MISSING_SEMICOLON, UNMATCHED_LBRACE, UNMATCHED_RBRACE, UNMATCHED_LPAREN, ASSIGNMENT_IN_CONDITION)
- Categories: 31
- Difficulty levels: 3 (beginner, intermediate, advanced)

**Output files:** `dataset.json`, `dataset.csv`, `token_sequences.txt`

### Week 9 — AI Model Training (`model_trainer.py`)
Trained and evaluated four ML models on the dataset.

**Feature extraction:**
- TF-IDF n-grams (500 features) on token sequences
- 13 structural features (brace balance, semicolon count, etc.)
- Total: 513 features per sample

**Models trained:**

| Model | Training Accuracy | CV Score | Verdict |
|---|---|---|---|
| Logistic Regression | 88.00% | 60.00% | Best |
| Random Forest | 100.00% | 33.33% | Overfit |
| Gradient Boosting | 100.00% | 33.33% | Overfit |
| SVM | 72.00% | 40.28% | OK |

**Best model:** Logistic Regression (CV = 60%)

### Week 10 — AI Integration (`main.py`, `run_compiler.py`, `ai_assistant.py`)
Integrated the C compiler with the Python AI layer.

**How it works:**
1. `main.py` receives source file as argument
2. `run_compiler.py` runs the compiled C compiler via subprocess
3. C compiler outputs JSON errors to stdout
4. `ai_assistant.py` classifies errors using rule-based engine
5. Plain English explanation shown with source code markers

**Error explanations cover:** MISSING_SEMICOLON, UNMATCHED_LBRACE, UNMATCHED_RBRACE, UNMATCHED_LPAREN, UNMATCHED_RPAREN, ASSIGNMENT_IN_CONDITION

### Week 11 — Optimization and Security (`optimization_module.py`)
Analyses C source code for inefficient patterns and security vulnerabilities.

**Optimization checks:**
| Check | Severity | Description |
|---|---|---|
| NESTED_LOOP | HIGH | O(n²) complexity warning |
| REPEATED_COMPUTATION | MEDIUM | Function calls inside loops |
| UNUSED_VARIABLE | LOW | Variables declared but never used |
| MAGIC_NUMBER | LOW | Hard-coded numbers |
| REDUNDANT_CONDITION | LOW | Unnecessary boolean comparisons |
| UNNECESSARY_RETURN | LOW | Redundant return at end of void function |

**Security checks:**
| Check | Severity | CWE |
|---|---|---|
| UNSAFE_FUNCTION | HIGH/CRITICAL | CWE-119 |
| HARDCODED_CREDENTIAL | CRITICAL | CWE-798 |
| NULL_DEREF | MEDIUM | CWE-476 |
| INTEGER_OVERFLOW | HIGH | CWE-190 |
| OUT_OF_BOUNDS | CRITICAL | CWE-125 |

**Unsafe functions flagged:** `gets()`, `strcpy()`, `strcat()`, `sprintf()`, `scanf()`, `system()`

### Week 12 — Testing and Validation (`test_cases.py`)
Full test suite covering all system components.

**Test categories:**
- Lexer tests — token generation and classification
- Parser tests — syntax error detection and AST building
- AI assistant tests — error explanation generation
- Integration tests — end-to-end pipeline
- Optimization tests — optimization hint detection
- Security tests — security issue detection
- Dataset tests — dataset structure and content validation
- Model tests — ML model loading and prediction

## System Architecture

```
Source Code (.c file)
        ↓
   Lexer (Lexer.c)
   reads char by char → token stream
        ↓
   Parser (Parser.c)
   checks grammar → AST + JSON errors
        ↓
   run_compiler.py
   reads JSON output from compiler
        ↓
   ai_assistant.py
   rule-based classification → explanation
        ↓
   main.py
   displays plain English output + source markers
        ↓
   optimization_module.py  (separate analysis)
   optimization hints + security warnings
```

## Known Limitations

- Mini-C subset only — not full ISO C standard
- No pointer support
- No struct support
- No preprocessor directives
- Dataset limited to 50 samples — larger dataset would improve ML accuracy
- ML model currently bypassed due to feature mismatch — rule-based classification used
